# FiberSocial — Project Instructions

## Git Workflow

- **Never push directly to `main`.** All changes — including documentation, formatting, or one-line fixes — must go through a feature branch and a pull request.
- **Never edit files in the primary checkout to start new work — always create an isolated worktree first.** Before touching any file, run `git worktree add ../FiberSocial-<slug> -b <type>/<slug>` and do the work there. This isn't optional even for "just config" changes: the primary checkout routinely has in-progress uncommitted edits sitting in it, and editing in place mixes unrelated changes into the same diff, making it impossible to cut a clean single-purpose PR without dragging along someone else's WIP (or reverting it by accident).
- **Never merge a PR yourself.** After creating or updating a PR, hand the URL to the user and wait. Never run `gh pr merge` or any equivalent.
- Branch naming: `feat/`, `fix/`, `chore/`, `docs/` prefixes matching the commit type.

## Secrets

- Never commit `ravelry.client_secret`. It lives in the gitignored `local.properties` as `ravelry.client_secret` for local builds, and in the `RAVELRY_CLIENT_SECRET` GitHub Actions repo secret for CI builds (`android-build.yml` and `release.yml` materialize it into CI's own `local.properties` — those injections are sanctioned, not leaks).
- Same pattern for release signing: the keystore and its passwords live in `local.properties`/a gitignored keystore file locally, and in the `RELEASE_KEYSTORE_BASE64`, `RELEASE_KEYSTORE_PASSWORD`, `RELEASE_KEY_ALIAS`, `RELEASE_KEY_PASSWORD` GitHub Actions repo secrets for CI (`android-build.yml` decodes the base64 secret back into a keystore file and writes the rest into CI's `local.properties`). Unlike the Ravelry secrets, these are optional in `android-build.yml` — CI skips release signing/build entirely (not a failure) when `RELEASE_KEYSTORE_BASE64` isn't set. The tag-triggered `release.yml` does the same materialization but treats all of these as required, failing loudly if any is missing.

## Building & running the Android app

- `local.properties` is gitignored, so a fresh `git worktree add` checkout won't have it. Copy it from an existing checkout (`src/platform/android/local.properties`) before building, or OAuth login will fail with `invalid_client` (empty `ravelry.client_id`/`ravelry.client_secret`).
- `RAVELRY_CLIENT_ID`/`RAVELRY_CLIENT_SECRET` are `BuildConfig` fields generated from `local.properties`. Because they're compile-time-constant `String`s, Kotlin inlines their value into every call site at compile time — if you add/fix `local.properties` *after* an earlier build already ran without it, a plain `./gradlew assembleDebug` can report everything as `UP-TO-DATE` and skip recompiling those call sites, silently keeping the stale (empty) inlined value even though the generated `BuildConfig.java` itself looks correct. If auth fails with an empty `client_id` despite correct `local.properties`, run `./gradlew clean` before rebuilding.
- A corrupted on-device `EncryptedSharedPreferences` keyset (`AndroidKeysetManager`/Tink errors, e.g. `InvalidProtocolBufferException: Protocol message contained an invalid tag` — can happen after an uninstall/reinstall cycle) is self-healed by the app since `encryptedKeyValueStore` gained recovery: it wipes the unreadable prefs file and recreates it, so instead of crashing on launch the app just asks the user to log in again. If a build predating that fix crashes this way, recover manually with `adb shell pm clear com.myhobbyislearning.fibersocial`. (After a debug↔release build-type switch `deploy.sh` uninstalls the old build; the reinstall then self-heals on next launch, so it no longer needs a `pm clear` step.)

## Building & running the iOS app

- The Xcode project is `src/platform/ios/FiberSocial.xcodeproj`; its "Build ComposeApp framework" phase runs `./gradlew :composeApp:embedAndSignAppleFrameworkForXcode` in `src/platform/android`, so Kotlin changes rebuild automatically. The script validates JDK candidates by major version (an inherited `JAVA_HOME` may point at an old JDK; it probes `java_home -v 17` and homebrew's `openjdk@17` as fallbacks).
- Secrets mirror Android's `local.properties`: put `RAVELRY_CLIENT_ID`/`RAVELRY_CLIENT_SECRET` in the gitignored `src/platform/ios/Config.local.xcconfig` (`Config.xcconfig` holds empty committed defaults and optionally includes it). Without them the app runs but login fails with `invalid_client`, with a launch-time warning.
- Simulator build/test from the CLI: `xcodebuild -project FiberSocial.xcodeproj -scheme FiberSocial -destination 'platform=iOS Simulator,name=iPhone 17' build` (or `test` — the hosted XCTests cover the Keychain/NSUserDefaults stores; the bare Kotlin/Native test runner has no keychain daemon, so `:composeApp:iosSimulatorArm64Test` keychain write tests skip themselves there by design).
- Feature-complete vs Android (#119): device photo uploads via PHPicker (HEIC transcodes to JPEG on upload), app icon, platform-aware settings copy. Notifications (#118): reminders are OS-scheduled local notifications; new-event polling is a sync on every foreground activation plus opportunistic `BGAppRefreshTask` (needs a real device to observe — the simulator rejects background-task submission, logged and harmless). TestFlight steps: src/platform/ios/README.md; device verification: docs/ios-device-checklist.md.

## Release builds

- `./gradlew assembleRelease` (or `./deploy.sh --release`) needs a signing keystore, configured the same way as the Ravelry secrets: gitignored, read from `local.properties` at build time. `app/build.gradle.kts` only creates the `release` signing config if `release.store.file` is set, so builds without it stay unsigned rather than failing.
- `./gradlew bundleRelease` builds the Android App Bundle (`.aab`) Play Console requires for store submissions — `deploy.sh`/direct-install users still want the APK from `assembleRelease`. It shares the same `release` signing config and `local.properties` keys automatically (the Android application plugin wires `bundleRelease` to whatever `signingConfig` is set on the `release` buildType); no separate configuration exists for it. Output: `app/build/outputs/bundle/release/app-release.aab`.
- Required `local.properties` keys: `release.store.file` (path relative to `src/platform/android/`), `release.store.password`, `release.key.alias`, `release.key.password`.
- The actual keystore file (`*.keystore`/`*.jks`, gitignored) doesn't come along with `git worktree add` any more than `local.properties` does — copy it into a fresh worktree from an existing checkout alongside `local.properties`, don't regenerate it (a new keystore produces a different signature and can't upgrade-install over an app signed by the old one).
- CI (`android-build.yml`) builds and uploads a signed `app-release` artifact on every push/PR too, once the `RELEASE_KEYSTORE_BASE64` repo secret (and its companion password/alias secrets) are set — see the Secrets section above. This is only a transient (7-day) Actions artifact for early smoke-testing, not a public release — see "Versioning & cutting a release" below for that.

## Versioning & cutting a release

- Versioning is manual, not automatic per-commit: pick a `MAJOR.MINOR.PATCH` version, then run `scripts/release.sh <version>` (e.g. `scripts/release.sh 1.2.0`) from `main` with a clean working tree, up to date with `origin/main`. It tags the current commit `vMAJOR.MINOR.PATCH` and pushes the tag.
- Pushing a `v*.*.*` tag triggers `.github/workflows/release.yml`, which builds the signed release APK and AAB and publishes both as assets on a GitHub Release (title = the tag, notes auto-generated from merged PRs since the previous tag via `gh release create --generate-notes`), marked as the repo's latest release. The AAB is for uploading to Play Console; it isn't directly installable via `adb install`/sideloading the way the APK is.
- `app/build.gradle.kts` derives `versionName`/`versionCode` from the tag when the current commit is exactly on one (`git describe --tags --exact-match --match 'v[0-9]*.[0-9]*.[0-9]*'`): `versionName = "MAJOR.MINOR.PATCH"`, `versionCode = MAJOR * 1_000_000 + MINOR * 1_000 + PATCH`. This needs to strictly increase release-over-release for in-place upgrade installs (tapping a newly downloaded APK) to work — don't tag a lower version after a higher one has shipped. Ordinary (non-tagged) builds are unaffected: they keep the existing git-hash `versionName` and `versionCode = 1`. One local trap: the derivation keys off the *commit*, not the build type, so even a debug `deploy.sh` build made while HEAD sits on a release tag gets the big tag-derived `versionCode` — the next dev build (back to `versionCode = 1`) is then an install "downgrade", which is why `deploy.sh` passes `adb install -d`.
- The public download link is always `https://github.com/torrey1028/FiberSocial/releases/latest/download/app-release.apk` — GitHub resolves `releases/latest` to whichever release was most recently marked latest, so this automatically follows whatever you most recently tagged and pushed. It does **not** update on every push to `main` — only on a release tag push, unlike the interim setup that preceded this one.

## Logging

- Use `println("FiberSocial: ...")` for debug logging in common module code (visible in logcat as `System.out` tag).
- Rely on logcat for debugging — do not use screenshots as a substitute.
